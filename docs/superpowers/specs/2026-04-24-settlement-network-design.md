# AI局沙盘聚落网络扩展设计

## 背景

当前主设计文档中的地图结构以“大城市”为核心，地图节点数量偏少。  
这会带来几个直接问题：

- 地图过小，大城市之间的战略纵深不足。
- 行军路径过短，缺少逐段推进的战争节奏。
- 攻城战容易变成“大城市之间直接对撞”，缺少外围争夺和缓冲地带。
- 武将 AI 缺少对中间节点、补给链和外围控制的决策空间。

因此需要在现有地图系统上引入 **城镇层**，让地图从“少量大城市点对点连接”扩展为“城市与城镇共同组成的连续聚落网络”。

---

## 目标

本设计需要满足以下目标：

1. 让地图规模明显扩大，但不引入第三套完全不同的规则。
2. 大城市和城镇采用同一类核心规则，只在规模和价值上存在差异。
3. 两座大城市之间允许存在多个中间城镇，形成连续路径。
4. 路网既保留 LLM 生成的历史叙事感，也能通过系统校验保证可玩性。
5. 行军、占领、补给、AI 决策都必须感知城镇网络的存在。

---

## 方案概述

采用 **统一聚落节点 + 混合路网生成模型**。

### 1. 统一聚落节点

地图上的所有地点统一建模为“聚落节点”，仅通过 `settlement_type` 区分：

- `city`
- `town`

两者共享同一套核心行为规则：

- 可被占领
- 可驻军
- 可补给
- 可征兵
- 可作为战斗发生地
- 可作为行军路径节点

两者的差别只体现在默认模板值上：

- `city`：更大、更富、更坚固、战略价值更高
- `town`：更小、更密集、单点价值较低，但在路径控制和补给链上作用显著

### 2. 连续聚落路网

地图不再采用“大城市之间直接连接”的简化结构，而是允许形成如下连续链路：

```text
许昌 <-> A镇 <-> B镇 <-> 洛阳
```

因此路网允许以下连接类型：

- `city <-> city`
- `city <-> town`
- `town <-> town`

但连接关系必须满足基本地理合理性，不能随意乱连。

### 3. 混合路网生成

路网生成采用两阶段：

1. LLM 生成聚落、坐标、地形、初始归属和建议连接
2. 系统根据规则进行校验、补边、裁边和修正

这样既保留世界叙事感，也避免出现断路、过密连接和明显失真的地图。

---

## 数据模型调整

### 1. 原 `cities` 扩展为 `settlements`

建议将原来的 `cities` 表扩展或重构为更通用的 `settlements`：

```text
settlements
  id
  name
  settlement_type
  x
  y
  terrain
  owner
  defense_level
  recruit_capacity
  food_output
  strategic_value
```

字段说明：

- `settlement_type`：`city` 或 `town`
- `defense_level`：防御等级
- `recruit_capacity`：征兵能力
- `food_output`：粮草产出能力
- `strategic_value`：战略价值评分

若当前阶段不希望重构表名，也可以保留 `cities`，但字段上必须支持区分 `city` 与 `town`。  
从长期清晰性看，推荐直接使用 `settlements`。

### 2. 驻军仍可沿用统一模型

驻军表可继续使用统一结构：

```text
garrisons
  general_id
  settlement_id
  troops
  food
```

这里不需要为城镇单独设计驻军表。  
只要 `settlement_id` 能同时指向城市和城镇即可。

### 3. 图结构调整

`graph.json` 或图结构存储中，连接关系应从“城市连接”扩展为“聚落连接”。

示例：

```python
("xuchang", "connects", "town_a", {"distance": 2})
("town_a", "connects", "town_b", {"distance": 1})
("town_b", "connects", "luoyang", {"distance": 2})
```

此时图中的节点不再默认都是大城市，而是完整的聚落网络节点。

## 初始化生成规则

### 1. LLM 负责生成的内容

`init_scenario.py` 在初始化世界时，LLM 需要返回以下结构：

- 聚落列表：城市与城镇
- 每个聚落的类型、名称、坐标、地形、归属、规模说明
- 建议连接关系
- 哪些城镇位于哪些大城市之间或周边

示意：

```json
{
  "settlements": [
    { "id": "xuchang", "name": "许昌", "type": "city", "x": 0, "y": 0, "terrain": "平原", "owner": "魏" },
    { "id": "town_a", "name": "长社", "type": "town", "x": 2, "y": 0, "terrain": "平原", "owner": "魏" },
    { "id": "town_b", "name": "偃师", "type": "town", "x": 4, "y": 0, "terrain": "平原", "owner": "汉" },
    { "id": "luoyang", "name": "洛阳", "type": "city", "x": 6, "y": 0, "terrain": "平原", "owner": "汉" }
  ],
  "connections": [
    { "from": "xuchang", "to": "town_a", "distance": 2 },
    { "from": "town_a", "to": "town_b", "distance": 1 },
    { "from": "town_b", "to": "luoyang", "distance": 2 }
  ]
}
```

### 2. 系统负责校验和修正的内容

LLM 输出后，系统执行第二阶段图校验：

1. 去除重复边、自环和非法引用
2. 检查整体连通性，防止出现孤岛节点
3. 检查是否存在明显不合理的超长边
4. 根据坐标补齐必要缺边，避免地图断裂
5. 对过密区域适度裁边，避免形成毫无战略意义的蜘蛛网

### 3. 城镇数量规则

每个大城市周边的城镇数量采用可配置动态生成，默认建议：

- 小型区域：2 个城镇
- 中型区域：3 个城镇
- 核心区域：4 个及以上城镇

具体数量由剧本规模、地理密度和战略重要性共同决定，而不是写死。

## 行军系统调整

### 1. 从点对点行军改为路径行军

原设计中，行军更接近“城市到城市直接跳转若干天”。  
引入城镇网络后，行军必须改为按路径逐段推进。

例如：

```text
许昌 -> 长社 -> 偃师 -> 洛阳
```

系统应先计算从起点到目标的路径，再按节点顺序推进。

### 2. 每段行军都可触发事件

军队每经过一个聚落节点或一段边时，都应触发事件检查，例如：

- 遭遇敌军
- 粮草不足
- 城镇投降
- 敌军截断补给
- 天气变化导致延迟

这样战争不再是“出发后直接到终点”，而是带有过程性的推进。

### 3. 中断式推进

若路径中某个节点出现以下情况，行军应中断：

- 节点被敌军重兵占据
- 发生遭遇战
- 粮草不足以继续推进
- 玩家或 GM 改变命令

这会让地图真正参与战争过程，而不是仅作为战斗背景。

---

## 占领与控制规则

### 1. 城镇与城市共享占领规则

城镇和大城市都应支持：

- 被进攻
- 被占领
- 驻军防守
- 提供补给
- 作为部队集结点

因此“城镇”不是纯装饰节点，而是完整的战略单位。

### 2. 默认差异模板

推荐默认差异如下：

- `city`
  - 更高防御等级
  - 更高驻军上限
  - 更高粮草与征兵产出
  - 更高战略价值

- `town`
  - 中低防御等级
  - 中低驻军上限
  - 中低粮草与征兵产出
  - 较低单点价值，但更高网络价值

“网络价值”指的是：一个城镇虽然本身不如大城市重要，但可能控制着两座核心城市之间的通道。

### 3. 交通与补给链意义

占领城镇的意义不只是多一个据点，还包括：

- 打通前进路线
- 切断敌军补给链
- 提供中途驻扎与补粮点
- 作为围攻大城市前的外围跳板

因此很多战役将不再是“直接打大城市”，而是“先夺取路上的若干关键城镇”。

## AI 决策影响

### 1. 行军目标更加分层

加入城镇网络后，武将不应只考虑“是否攻打洛阳”，而应考虑：

- 是否先夺取中间城镇
- 是否切断敌军主通道
- 是否绕路走较弱防线
- 是否先占外围节点再围大城

### 2. 防守决策更加立体

防守方也不再只是“守城”：

- 可以派兵驻守关键镇口
- 可以放弃外围小镇，收缩防线
- 可以利用中间节点拖慢敌军推进
- 可以在城镇链路上设置伏击或延滞

### 3. 补给与粮草决策更合理

因为路径更长、节点更多，AI 在判断是否出兵时应更重视：

- 当前粮草是否足够走完整条路线
- 沿线是否有己方补给点
- 是否会被敌人切断中途节点

这会让战争节奏更自然，也更符合你希望的“地图大了之后战斗更精彩”的目标。

---

## 战斗系统影响

### 1. 战斗地点不再只发生在大城市

未来战斗可以发生在：

- 大城市攻防战
- 城镇争夺战
- 两个聚落之间的遭遇战

这会显著增加战斗类型。

### 2. 战报内容更丰富

战报可以自然体现推进链条，例如：

- “刘备先夺长社，再破偃师，兵锋逼近洛阳”
- “曹操弃守外围城镇，退回许昌坚守”
- “黄巾军在中途小镇断粮，被迫撤军”

相比只有大城市，地图网络会让叙事更有层次。

---

## 对现有模块的影响

### 1. `init_scenario.py`

需要从“生成城市”升级为“生成完整聚落网络”。

### 2. `view.py`

需要支持展示：

- 城市与城镇类型
- 聚落之间的连接
- 关键通道

### 3. `time_engine.py`

需要支持：

- 按路径逐段推进
- 中途节点事件检查
- 行军中断与重规划

### 4. `battle.py`

需要支持更多战斗地点类型，但不一定需要重写核心结算逻辑。  
主要改动在于战斗上下文中加入：

- 当前战斗地点类型
- 当前是否为通道节点
- 当前是否涉及补给线争夺

### 5. `state.py`

需要新增对 `settlements` 及其连接关系的查询与更新能力。

---

## 失败场景与处理

### 1. 地图过密

若系统补边过多，可能导致任何地点都可快速到达。  
处理方式：

- 设置每个节点最大连接数
- 对过长且重复意义低的边优先裁剪

### 2. 地图过稀

若连接过少，会出现断路或路线单一。  
处理方式：

- 强制整体连通
- 对关键区域补充中间节点连接

### 3. 城镇过多导致信息负担过高

若节点数量爆炸，玩家阅读和 AI 规划成本都会显著上升。  
处理方式：

- 按剧本规模动态控制总节点数
- 默认只突出显示关键聚落和关键通道

## 测试重点

需要覆盖以下测试：

1. 初始化时能生成同时包含城市和城镇的地图
2. LLM 输出异常连接时，系统能正确校验修复
3. 地图整体连通，无孤立核心区域
4. 行军能沿节点路径逐段推进
5. 中间节点被敌军占据时，行军能正确中断
6. 城镇可被占领、驻军、补给
7. AI 能在目标决策中考虑中间聚落
8. 战报可正确描述多节点推进过程

---

## 推荐落地顺序

建议按以下顺序落地：

1. 扩展地点模型，支持 `city` / `town`
2. 扩展初始化流程，生成聚落网络
3. 完成路网校验与修正逻辑
4. 改造行军系统为路径推进
5. 补齐城镇占领、驻军、补给规则
6. 最后再增强 AI 决策与战报表达

这样可以先把地图结构做对，再逐步让战斗和叙事吃到收益。

---

## 结论

为了让地图更大、战争更有推进层次，应将地图从“少量大城市节点”升级为“城市 + 城镇组成的连续聚落网络”。  
在这个模型中：

- 大城市和城镇共享核心规则
- 两者只在规模和价值上区分
- 大城市之间允许通过多个城镇连接
- 路网采用“LLM 生成建议 + 系统校验修正”的混合机制

这套设计能显著增强地图纵深、战斗叙事和 AI 战略决策空间，是现有主规格的自然扩展方向。
