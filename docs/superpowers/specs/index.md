# AI局沙盘规格索引

## 文档结构

当前规格采用“1 份主规格 + 2 份专题补充”的组织方式，避免把所有扩展能力都堆叠进单一文档，导致阅读成本和实现边界失控。

## 文档列表

### 1. 主规格

- [2026-04-23-three-kingdoms-game-design.md](./2026-04-23-three-kingdoms-game-design.md)
- 作用：定义 AI局沙盘 的整体目标、Hermes 架构、基础游戏循环、Agent 设计、时间事件系统、战斗系统、剧本初始化与微信接入方式

### 1.1 第一版可运行版本设计

- [2026-04-24-first-runnable-version-design.md](./2026-04-24-first-runnable-version-design.md)
- 作用：定义空项目如何落成第一版可运行骨架，明确 CLI 主链、Hermes 建局依赖、存档结构、错误处理与后续接入边界

### 2. 玩家接管扩展

- [2026-04-24-player-takeover-design.md](./2026-04-24-player-takeover-design.md)
- 作用：定义中途玩家加入、接管现有武将、在线时玩家控制、离线后回退 AI 托管的控制权模型

### 3. 聚落网络扩展

- [2026-04-24-settlement-network-design.md](./2026-04-24-settlement-network-design.md)
- 作用：定义地图从“大城市节点”扩展为“城市 + 城镇连续聚落网络”，以及由此带来的路网、行军、占领与 AI 决策变化

## 推荐阅读顺序

1. 先读主规格，建立整体认知
2. 再读第一版可运行版本设计，明确最小落地范围与项目骨架
3. 再读玩家接管扩展，理解多人接入和控制权切换
4. 最后读聚落网络扩展，理解地图规模扩展与路径战争

## 实现建议

后续写实现计划时，建议按以下顺序分解：

1. 先按“第一版可运行版本设计”落地项目骨架
2. 再以主规格为基线扩展核心玩法 MVP
3. 将“玩家接管扩展”作为多人接入子项目
4. 将“聚落网络扩展”作为地图与行军子项目

这样可以避免三份规格并行落地时相互牵制，也更适合逐步实现与验证。
