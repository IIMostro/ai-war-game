# AI War Game 第一版可运行版本设计

## 背景

当前项目仍处于空仓库脚手架阶段：

- `main.py` 仅输出一行示例文本
- 项目尚未形成稳定的 CLI 命令入口
- 还没有可用的游戏存档结构与应用层边界
- Hermes/LLM 虽然是目标架构的一部分，但当前没有真正接入到“建局”主链

因此，第一版设计的首要目标不是扩展玩法，而是先让项目以正确的方向“运行起来”。

这里的“运行起来”不是指程序能打印一句话，而是指用户可以在本地启动一个可交互的 CLI，完成以下最小闭环：

1. 创建一局新游戏
2. 查看当前游戏状态
3. 执行一条玩家命令
4. 从本地存档继续操作

同时，第一版必须坚持项目的长期方向：

- 剧本初始化必须通过 Hermes/LLM 生成
- 当 Hermes 运行环境不满足时，系统直接失败并给出明确诊断
- 微信/Hermes 外部接入虽然不是本版交付重点，但必须在架构层预留清晰边界

---

## 目标

第一版设计需要满足以下目标：

1. 让空项目变成一个真正可运行、可验证、可持续演进的 CLI 游戏骨架。
2. 打通 `建局 / 查看状态 / 执行命令 / 存档恢复` 的最小主链。
3. 将 Hermes/LLM 纳入建局流程，而不是先做静态示例世界。
4. 在环境缺失时快速失败，避免用户进入半可用状态。
5. 为后续微信入口、Hermes Skill、更多玩法系统保留稳定的应用层接口。

---

## 非目标

第一版明确不包含以下内容：

- 多 Agent 并发决策编排
- 完整战斗系统
- 完整内政、外交、天气、时间推进系统
- 微信真实接入联调
- 复杂地图网络与聚落扩展落地
- 玩家中途接管与控制权租约

这些能力可以在后续专题设计中继续扩展，但不应阻塞“第一版先可运行”。

---

## 方案选择

本次设计比较了三种方向：

### 方案 A：单进程 CLI + 明确服务分层 + 外部 Hermes 适配器

做法：

- 使用单一 Python CLI 入口作为本版正式入口
- 将业务拆分为 CLI、应用层、领域层、持久化层和 Hermes 适配层
- 建局时通过 Hermes/LLM 生成剧本
- 存档以本地结构化文件保存
- 后续微信/Hermes 外部入口复用同一应用层

优点：

- 最符合“先跑通，再扩展”的节奏
- 不会把业务逻辑绑死在 CLI 或外部命令调用上
- 后续接微信时无需重写核心流程

缺点：

- 第一版需要先定义清晰接口，前期设计成本略高于脚本堆叠方案

### 方案 B：命令脚本驱动 + 文件协议

做法：

- 将每个能力拆成独立脚本
- `main.py` 只负责调度
- 各脚本之间通过 JSON 文件交换数据

优点：

- 实现速度快
- 形式上接近未来 Hermes Skill 调脚本的使用方式

缺点：

- 在空项目阶段容易把代码组织成分散脚本集合
- 测试、调试和长期维护成本较高

### 方案 C：Hermes 优先的代理式入口

做法：

- 第一版就围绕 Hermes Skill 和代理入口组织系统
- CLI 仅作为附属包装

优点：

- 最接近最终目标形态

缺点：

- 在当前仓库仍为空项目时风险最高
- 会把“先运行起来”变成“先完成较重的 Hermes 集成工程”

### 推荐方案

第一版采用 **方案 A：单进程 CLI + 明确服务分层 + 外部 Hermes 适配器**。

原因很直接：

- 它能最快形成真正可用的本地运行闭环
- 同时不会阻断长期的 Hermes/微信接入方向
- 相比把逻辑都塞进 `main.py`，这套分层更适合后续持续扩展

## 第一版系统边界

第一版只解决一件事：**让用户在本地 CLI 中创建并持续操作一局由 Hermes/LLM 生成的游戏世界**。

本版正式支持的用户能力只有四项：

- `new-game`：创建新游戏
- `show`：查看当前游戏状态
- `command`：执行一条玩家命令
- `list-saves`：查看本地已有存档

其中：

- `new-game` 是第一版最关键的入口，承担环境检查、剧本生成与世界初始化
- `show` 和 `command` 共同承担“恢复继续玩”的能力，因此本版不单独设计 `resume`
- `list-saves` 用于改善多局管理与恢复体验

本版入口分为两类：

- `CLI 入口`：第一版唯一正式可用入口
- `Hermes/微信入口`：本版只定义接口边界，不要求实际打通

---

## 模块划分

第一版建议使用以下模块边界。

### 1. `cli`

职责：

- 解析命令行参数
- 校验基础输入格式
- 格式化终端输出
- 设置退出码

约束：

- 不直接读写存档文件
- 不直接拼接 Hermes 提示词
- 不直接修改领域状态

### 2. `application`

职责：

- 编排核心用例
- 组织建局、查看、执行命令和列出存档等流程
- 定义稳定的输入输出数据结构

建议包含的用例服务：

- `CreateGameService`
- `ShowGameService`
- `ExecuteCommandService`
- `ListSavesService`

这一层是未来 CLI 与微信入口共同复用的主边界。

### 3. `domain`

职责：

- 定义核心游戏对象
- 承担第一版最小规则约束
- 表达状态变化结果

第一版建议至少包含以下对象：

- `GameSession`
- `Scenario`
- `PlayerIdentity`
- `WorldState`
- `CommandRequest`
- `CommandResult`

领域层不负责外部进程调用，也不负责文件存储。

### 4. `infrastructure.persistence`

职责：

- 管理存档目录
- 读写结构化游戏数据
- 处理版本字段和格式兼容问题

第一版优先采用文件型持久化，而不是立即引入数据库。

### 5. `infrastructure.hermes`

职责：

- 执行 Hermes 运行环境检查
- 调用 Hermes/LLM 生成初始剧本
- 解析和校验 Hermes 输出
- 将外部错误包装成应用层可处理的错误类型

第一版中，这一层只负责“建局生成剧本”，不提前承担多 Agent 编排。

### 6. `interfaces`

职责：

- 定义未来 Hermes/微信接入的消息适配边界
- 将外部消息统一转换为应用层请求对象

本版允许这里只有接口定义与说明，不要求真实实现。

## 数据流设计

### 1. 建局流程

`new-game` 的标准执行顺序如下：

1. CLI 接收并解析参数，例如剧本主题、玩家身份、存档名。
2. CLI 调用 `CreateGameService`。
3. `CreateGameService` 先调用 Hermes 健康检查。
4. 若检查失败，流程立即终止并返回明确错误。
5. 若检查通过，调用剧本生成器请求 Hermes/LLM 生成初始世界。
6. 应用层校验返回结果是否满足最小结构。
7. 校验通过后构造 `GameSession` 与 `WorldState`。
8. 持久化层写入存档目录。
9. CLI 输出建局成功摘要。

这个流程的关键要求是：**环境检查必须早于建局写盘**，避免产生半初始化存档。

### 2. 查看流程

`show` 的标准执行顺序如下：

1. CLI 接收存档标识。
2. 调用 `ShowGameService`。
3. 服务层从持久化层加载会话与世界状态。
4. 应用层组装展示摘要。
5. CLI 负责最终渲染。

### 3. 执行命令流程

`command` 的标准执行顺序如下：

1. CLI 接收存档标识与玩家命令文本。
2. 调用 `ExecuteCommandService`。
3. 服务层读取当前游戏状态。
4. 将输入封装为统一的 `CommandRequest`。
5. 领域层执行第一版支持的最小规则。
6. 返回 `CommandResult` 与更新后的状态。
7. 持久化层保存新状态并追加事件日志。
8. CLI 输出执行结果。

### 4. 存档列出流程

`list-saves` 的标准执行顺序如下：

1. CLI 调用 `ListSavesService`。
2. 服务层扫描存档目录。
3. 返回存档元数据摘要。
4. CLI 展示列表。

## 命令集设计

第一版 CLI 应采用显式子命令，而不是一开始就做自然语言统一入口。

建议命令集如下：

### `new-game`

输入：

- 剧本主题
- 玩家身份
- 存档名

行为：

- 检查 Hermes 环境
- 调用 LLM 生成剧本
- 初始化游戏世界
- 写入新存档

### `show`

输入：

- 存档名

行为：

- 展示当前日期
- 展示玩家身份与所属势力
- 展示主要世界摘要
- 展示最近事件

### `command`

输入：

- 存档名
- 玩家命令文本

行为：

- 解析并执行第一版支持的最小动作
- 更新世界状态
- 写入事件日志

### `list-saves`

输入：

- 无，或可选存档根目录

行为：

- 枚举已有存档
- 展示名称、创建时间、当前日期、玩家身份等摘要

## 第一版存档结构

第一版建议每局游戏使用一个独立目录，例如：

```text
data/saves/<save_id>/
```

目录内部至少包含以下文件：

### `session.json`

存放会话元数据，例如：

- `save_id`
- `created_at`
- `updated_at`
- `current_day`
- `player_identity`
- `scenario_summary`
- `schema_version`

### `world.json`

存放当前世界状态，例如：

- 势力信息
- 角色信息
- 聚落或城市摘要
- 玩家当前归属
- 资源与其他最小状态字段

### `events.jsonl`

使用追加式日志记录关键事件，便于：

- 调试命令执行结果
- 追踪世界演化过程
- 后续做回放或审计

### `scenario.json`

保留 Hermes/LLM 生成的原始剧本结果，用于：

- 校验建局质量
- 调试解析逻辑
- 未来支持重放或二次处理

### 为什么第一版不用数据库

当前项目仍处于空项目起步阶段，文件型存档更适合第一版：

- 可读性高，调试成本低
- 用户可直接检查生成结果
- 应用层与持久化层仍可通过接口隔离
- 后续迁移到 SQLite 时无需推翻上层结构

## Hermes 集成策略

第一版要求剧本必须通过 Hermes/LLM 生成，不提供静态示例世界作为正式路径。

### 1. 环境检查

在执行 `new-game` 前，必须显式检查至少以下内容：

- Hermes CLI 是否可执行
- 必要模型配置是否存在
- 建局所需环境变量或配置项是否可读取

任何检查失败都必须直接中止流程并输出明确错误说明。

### 2. 剧本生成

Hermes 适配层负责：

- 组装建局请求
- 调用外部命令或接口
- 获取原始生成结果
- 解析为第一版需要的最小世界结构

### 3. 输出校验

即使 Hermes 调用成功，也不能直接信任输出。

应用层必须校验至少以下结构是否存在：

- 玩家身份信息
- 世界摘要
- 至少一组势力或角色数据
- 初始日期或时间锚点

若结构不满足要求，则视为生成失败。

---

## 错误处理

第一版必须采用“快速失败、明确报错”的策略，不提供降级模式。

### 1. 环境失败

示例：

- 未安装 Hermes
- 配置缺失
- 模型不可用

处理策略：

- `new-game` 立即失败
- 明确指出失败检查项
- 不创建正式存档目录

### 2. 生成失败

示例：

- Hermes 返回空结果
- 输出不可解析
- 必填字段缺失

处理策略：

- 终止建局
- 输出解析失败原因
- 不写入正式可用存档

### 3. 存储失败

示例：

- 目录不可写
- 文件序列化失败
- 存档版本冲突

处理策略：

- 明确指出失败文件与步骤
- 保证不会伪装成建局成功

---

## Hermes / 微信接入预留接口

虽然第一版只交付 CLI，但必须提前定义统一请求模型，避免后续新增入口时重写业务层。

建议定义统一消息封装，例如：

```text
CommandEnvelope
  source
  player_id
  save_id
  command_text
  timestamp
```

字段说明：

- `source`：消息来源，例如 `cli` 或 `wechat`
- `player_id`：当前操作用户标识
- `save_id`：目标存档
- `command_text`：原始命令文本
- `timestamp`：消息时间

CLI 入口负责把命令行输入转换成这个结构；未来微信入口负责把消息事件也转换成同一结构。  
应用层只接收统一请求，而不感知请求来自终端还是聊天平台。

这意味着后续新增微信入口时，只需要新增适配器，而不是修改核心用例服务。

## 测试策略

第一版虽然范围有限，但必须让“可运行”具备可验证性。

### 1. 单元测试

覆盖对象：

- 领域对象构造与最小规则
- 应用层输入输出组装
- 错误类型包装
- 存档元数据处理

### 2. 集成测试

覆盖链路：

- `new-game`
- `show`
- `command`
- `list-saves`

测试中不要求依赖真实 Hermes 环境，而是通过抽象接口注入假响应，以验证：

- 应用层编排是否正确
- 持久化层是否稳定
- 错误流是否可预测

### 3. 手工验收

在真实本机环境中，至少执行一次完整链路：

1. 运行 `new-game`
2. 确认环境检查通过
3. 确认剧本成功生成
4. 确认存档目录生成
5. 运行 `show`
6. 运行一次 `command`
7. 再次运行 `show` 验证状态变化已保存

---

## 第一版验收标准

第一版完成后，至少应满足以下标准：

1. 用户能通过单一 CLI 入口看到清晰帮助并执行 `new-game`。
2. `new-game` 在 Hermes 环境不满足时会立即失败，并输出明确诊断。
3. `new-game` 在环境满足时能生成一局有效存档，并保留原始剧本结果。
4. 用户能通过 `show` 和 `command` 对同一存档持续操作。
5. 未来新增 Hermes/微信入口时，不需要修改领域层与应用层主接口。

---

## 后续演进建议

当第一版可运行骨架稳定后，后续规划建议按以下顺序推进：

1. 扩展 `command` 的领域动作集合，形成更真实的最小玩法循环。
2. 将时间推进与事件日志结合，形成基础世界演化能力。
3. 在应用层引入多角色决策编排，再扩展到多 Agent。
4. 接入真实 Hermes/微信消息入口，复用同一应用服务。
5. 根据数据复杂度决定是否从文件存档迁移到 SQLite。

---

## 结论

第一版最重要的不是做出完整游戏，而是建立一条真实、稳定、可验证的主链：

- 通过 CLI 建局
- 通过 Hermes/LLM 生成剧本
- 将世界持久化到本地
- 能查看状态
- 能执行命令
- 能持续在同一局上继续操作

只要这条主链成立，项目就不再是一个空壳脚手架，而是一个可以继续向 AI 沙盘战争游戏演进的有效基础。
